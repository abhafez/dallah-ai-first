Overview

The Client Admin Dashboard is a lightweight, web-based interface designed for client administrators managing corporate trainee enrollment on our LMS platform. The dashboard addresses the need for a non-technical workflow to handle common onboarding and management actions—including enrolling single and multiple trainees, updating enrollment assignments (including deleting enrollment), modifying user details, and viewing course attendance progress. Previously, these tasks required manual API calls using tools like Postman, The new dashboard enables secure self-service interactions via a simple, guided UI.
Scope: Phase 1
Included Features

    Send Single User: Form-based creation of individual trainees, requiring user national ID, mobile number (must be a valid Saudi number), course (selected from a dropdown of course titles), language, branch (selected from a dropdown of branch names). The mapping between course/branch titles and LMS IDs is configured by the internal team in the backend. If a selected course or branch is not configured, show a clear error instructing the admin to contact support.

    Send Bulk Users: Upload and process multiple user records at once (using CSV upload). CSV must include user ID, mobile (Saudi number), course title, and branch name. If any row references an unmapped course or branch, that row fails with a clear error; valid rows should be processed and failures reported per-row.

    Update Enrollment (Replace and Delete): Ability to replace a user’s enrollment with a new course (effectively deleting the current enrollment and adding the new one), and to delete enrollment directly.

    Update User Profile: Edit trainee details (name, user ID, mobile) with uniqueness constraints and real-time sync to backend and Keycloak.

    View Progress Notifications (Attendance): Read-only display of users who have started courses, based on LMS progress notification API.

Configuration
User Flows

1.  Login

    Happy Path: Admin accesses dashboard, enters credentials, and lands on the dashboard home.

    Error Scenarios: Incorrect password, inactive account → display clear, actionable error.

2.  Send Single User

    Happy Path: Admin selects “Add User”, enters all required fields (user ID, course, mobile, branch), and submits. Success message shown if accepted.

    Errors:

        Missing mandatory fields: show inline error highlighting required fields.

        Duplicate user (by ID or mobile): show error that user already exists.

        Backend/API failure: show general error message and prompt to retry.

        Course or branch doesn’t exists in mapping configuration.

3.  Send Bulk Users (CSV Upload)

    Happy Path: Admin downloads CSV template (if available), fills required columns, uploads file. System processes each row and reports results (success count, per-user issues).

    Errors:

        File format invalid: alert user to upload in correct format.

        Missing/invalid fields in rows: summarize errors per-user after upload (option to download error report).

        Partial failures: proceed with valid users, clearly mark failed entries.

        Course or branch doesn’t exists in mapping configuration.

4.  Update Enrollment (Replace and Delete)

    Happy Path: Admin enters or searches for user by name or mobile or id, selects user, updates course or level or language or vehicle, and confirms. Enrollment can be replaced or deleted as needed.

    Errors:

        User not found: display error message.

        No current enrollment to replace or delete: display error.

        Required fields missing: highlight missing data.

5.  Update User Profile

    Happy Path: Admin locates user ), edits name, ID, or mobile, and submits. System enforces uniqueness for ID and mobile, and updates backend and Keycloak.

    Errors:

        Attempt to set duplicate ID/mobile: show specific error.

        Backend rejects update: present meaningful error.

6.  View Progress Notifications (Attendance)

    Happy Path: Admin opens "Attendance" or "Progress" view to see list of all users with “started” status and start date. Option to trigger manual refresh.

    Errors:

        API/load failure: show error with option to retry.

Functional Requirements
UI and Design Considerations

    Theme: Use an open-source or out-of-the-box admin dashboard theme. No custom branding required. Simple and unambiguous.

    Information Architecture: Keep actions clearly separated (e.g., “Add User”, “Import Bulk Users”, “Update Enrollment”, etc.).

    No Advanced Dashboards: Omit graphs, analytics, and advanced filtering; reserved for future phases.

Out of Scope for Phase 1

    Filtering, searching, exporting, or advanced reading of user or attendance data.

    Viewing or generating attendance reports—beyond simple started/not started indicators.

    Supporting create-only enrollment actions (outside of replace or delete).

    Multi-role or role-based permission controls; assume all users are client admins.

    UI customization or branding.

    Any feature not explicitly described in the phase 1 scope.

Success Criteria and Next Steps

Success Criteria:

    Client admins can execute all five core actions without developer intervention.

    Error messages are immediately visible and instructive, with clear recommended next steps.

    All data integrity and uniqueness requirements are met; no duplicate user records are created.

    No critical or blocker bugs are reported in production for any phase 1 workflow.
